# About Me

Section guideline for the **About Me** section, rendered on the resume as **"Professional Summary."** This file extends `formatStandards.md` — the universal rules there still apply; this file adds specifics. Where they conflict, `formatStandards.md` wins.

---

## 1. Purpose

The About Me section is a brief self-description: how the user works, their mindset and work habits, and their areas of strength. It sits near the top of the resume and is the first prose a recruiter reads — it must earn the rest of the read in a few seconds.

The rendered section heading is **"Professional Summary"** — not "About Me," which reads informal for a resume.

---

## 2. The AI's role

This is the first **genuinely generative** section — the AI composes prose, it does not just place or organize data.

- The AI writes the section from the user's Q&A answers about their work style, mindset, habits, and focus.
- It composes — but it **never invents.** Every trait, strength, or working style stated must come from the user's input (`formatStandards.md §9`). Do not attribute a mindset, value, or habit the user did not express.
- If the user's input is too thin to write a genuine summary, write a short honest version and raise a **note** suggesting they add more — never pad with generic filler.

---

## 3. Format

The section has two parts:

1. **A short paragraph** — 2-3 sentences, roughly 40-70 words. A cohesive description of who the user is professionally and how they work.
2. **A focus line** — a single line beginning `Focus areas:` followed by 3-5 keyword items separated by ` · ` (middot).

Example:

> Backend engineer with eight years across fintech and infrastructure. Breaks large problems into small, testable increments and prioritizes maintainable code over clever code. Communicates technical tradeoffs clearly between engineering and product teams.
>
> Focus areas: Distributed systems · Developer tooling · Code review

---

## 4. Voice

Per `formatStandards.md §1`, the section is written in **implicit-subject voice — never first person.**

- Never use `I`, `me`, `my`. About Me is the section most tempting to write in first person — do not.
- Correct: `Backend engineer specializing in distributed systems.`
- Wrong: `I am a backend engineer specializing in distributed systems.`

---

## 5. Avoid clichés — be concrete

About Me attracts more banned vocabulary than any other section. Personality clichés and generic adjectives (`formatStandards.md §5.1`, `§5.4`) are forbidden — replace every one with a concrete, specific statement of how the user actually works.

| Cliché (banned) | Concrete (preferred) |
|---|---|
| "Passionate, motivated developer" | "Backend engineer who favors readable code and incremental delivery" |
| "Strong team player, excellent communicator" | "Communicates technical tradeoffs clearly between engineering and product" |
| "Detail-oriented and results-driven" | "Catches edge cases early through systematic testing" |
| "Innovative problem solver" | "Breaks large problems into small, testable increments" |

The test: if a sentence could appear on any resume in any field, it is too generic. Every sentence should be something only this user could truthfully say.

---

## 6. Keywords

This section should carry key phrases for recruiters and ATS to pick up.

- Weave the user's actual **role, specialization, and core technologies** into the paragraph naturally — never as a list crammed into prose.
- The **focus line** is the deliberate keyword hit: 3-5 of the user's genuine specializations or strengths.
- Use canonical naming (`formatStandards.md §7`). Do not keyword-stuff — every keyword must reflect something the user actually does.

---

## 7. Metrics

If the user provided concrete figures — years of experience, scope, scale — they may appear here (e.g. "eight years across fintech"). Per `formatStandards.md §4`, never invent or estimate them.
